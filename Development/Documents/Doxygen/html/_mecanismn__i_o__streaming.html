I/O Streaming Mecanismn

Currently, there is no internal file formats handled by the library. Users have to write their own HOST producer (i.e. on CPU), by :

  • first, deriving from the GvCore::GvIProvider class,

  • then handle the common GvCore::GvIProvider::produceData() method.

    During the final Update Phase step of the GigaVoxels Pipeline, the Cache Mecanism asks the producer to provide its data by calling the GvCore::GvIProvider::produceData(). The producer receives useful information (nodes adreses, localization information, etc...) needed to load corresponding data file on disk.

    Examples can be found in the tutorials like, for instance, the Dynamic Load Tutorial Dynamic Load Tutorial.

File Formats and Organization on disk

Description

In this section, we describe how data are organized on disk in the Dynamic Load Tutorial Dynamic Load Tutorial. This is the common way of doing in all I/O streaming tutorials.

Files are named as follows : filename_BRxxx_Bxxx_Lxxx_Cxxx_type.extension,

where :

  • filename is the file name,
  • BRxxx : "BR" stands for "brick", "xxx" is the size in each dimension,
  • Bxxx : "B" stands for "border", "xxx" is the size of the bordel,
  • Lxxx : "L" stands for "level", "xxx" is the index of the level,
  • Cxxx : "C" stands for "channel", "xxx" is the index of the channel,
  • type : this is the type of the channel (i.e. uchar4, float, etc...)

  • extension : this is the file extension (see below for details)

    Files are organized in two groups :

  • node tiles information : file extension is .nodes,
  • and bricks of voxels : file extension is .bricks.

Example

The following screenshot shows the files associated to the 3D dragon model loaded in the Dynamic Load Tutorial Dynamic Load Tutorial.

Example of data files

In this example, files are named as follow :

  • fux : filename,
  • BR8 : each brick holds 8 voxels in each dimension, denoted as [8x8x8],
  • B1 : each brick has only one border on each side, so the real brick size is in fact [10x10x10],
  • Lxxx : there are 7 levels from the 0 coarser one to the more detailed level 6,
  • C0 : there is only one channel named C0,
  • uchar4 : the type of the the channel is a 4 dimensions unsigned char.

So,

  • fux_BR8_B1_L5.nodes is the file holding nodes information for level 5,
  • fux_BR8_B1_L5_C0_uchar4.bricks is the file holding bricks information for level 5.
